## 💡 서비스 소개

> **쿵합**은 한국 전통 사주 체계를 현대적으로 재해석한 소개팅 앱입니다.
> 생년월일과 태어난 시(時)를 기반으로 60갑자 중 하나의 **일주 동물**이 배정되고, 오행(五行) 궁합 점수로 인연을 찾습니다.

단순 프로필 매칭이 아니라, 수백 년의 사주 이론을 알고리즘으로 구현해 **나와 잘 맞는 사람**을 먼저 보여주는 것이 쿵합의 핵심입니다. 마음에 드는 상대에게 코인을 사용해 채팅방을 열고, 150개의 메시지를 나눈 뒤 서로 YES를 선택하면 비로소 연락처가 공개됩니다.

## 🔥 핵심 기능

### 🐯 일주 동물 배정
생년월일과 태어난 시를 입력하면 60갑자에서 나만의 일주 동물이 결정됩니다. 단순한 띠가 아닌, 일(日)의 천간·지지까지 반영한 더 정밀한 나의 사주 정체성입니다.

### 🔮 오행 궁합 점수
두 사람의 오행 관계(상생·비화·상극)를 기준으로 점수를 산출하고, 천간합·지지합·지지충 보정을 더해 0~100점의 궁합 점수를 계산합니다. 탐색 피드는 이 점수 순으로 정렬됩니다.

### 💬 코인 채팅
마음에 드는 상대에게 코인 1개를 사용해 채팅방을 엽니다. 150개의 메시지를 주고받은 뒤 양쪽 모두 YES를 선택해야 연락처가 공개됩니다. 한쪽이라도 NO거나 24시간 내 응답이 없으면 자동으로 종료됩니다.

### 📖 일주 동물 백과사전
60가지 일주 동물 각각의 특성과 궁합 해설을 제공합니다. 나의 일주 동물이 어떤 성향을 가지는지, 어떤 동물과 잘 맞는지 확인할 수 있습니다.

## 🔮 매칭 알고리즘

### 궁합 사전 계산

탐색 화면에서 궁합 점수를 실시간으로 계산하면 요청마다 전체 유저를 순회해야 하는 부담이 생깁니다. 쿵합은 이를 **가입 시점에 미리 처리**하는 방식으로 해결합니다.

신규 유저가 가입을 완료하면 `UserSignedUpEvent`가 발행되고, `@TransactionalEventListener(AFTER_COMMIT)`가 이를 받아 별도의 비동기 트랜잭션에서 기존 활성 유저 전체와의 궁합을 일괄 계산합니다. 결과는 `match_candidate` 테이블에 A→B, B→A 양방향으로 저장되어, 이후 탐색 API는 단순 조회만으로 정렬된 추천 목록을 반환할 수 있습니다.

`AFTER_COMMIT` 시점에 리스너가 실행되므로, 메인 트랜잭션이 롤백된 유령 유저와 궁합이 계산되는 문제가 원천 차단됩니다. 계산 중 오류가 발생하더라도 로그만 남기고 가입 응답에는 영향을 주지 않습니다.

### 궁합 점수 계산 (`CompatibilityCalculator`)

점수는 **오행 베이스 → 천간/지지 보정** 두 단계로 산출되며, 최종값은 `clamp(0, 100)`으로 범위를 고정합니다.

#### 1단계 — 오행(五行) 베이스

두 사람의 일주 오행 관계에서 기준 점수를 정합니다. 서로 에너지를 북돋는 **상생**이 가장 높고, 같은 기운인 **비화**, 서로를 억누르는 **상극** 순입니다.

| 관계 | 해당 조합 | 기준 점수 |
|------|-----------|----------|
| 상생(相生) | 목↔화, 화↔토, 토↔금, 금↔수, 수↔목 | 55 |
| 비화(比和) | 동일 오행 | 45 |
| 상극(相剋) | 나머지 | 25 |

#### 2단계 — 천간·지지 보정

일주의 천간과 지지를 추가로 대조해 점수를 보정합니다. 합(合)은 가산, 충(沖)은 감산입니다.

| 관계 | 해당 조합 | 보정 |
|------|-----------|------|
| 천간합(天干合) | 갑기 / 을경 / 병신 / 정임 / 무계 | +15 |
| 지지 육합(六合) | 자축 / 인해 / 묘술 / 진유 / 사신 / 오미 | +15 |
| 지지 삼합(三合) 쌍 | 인오 / 오술 / 인술 등 16쌍 | +10 |
| 지지 충(沖) | 자오 / 축미 / 인신 / 묘유 / 진술 / 사해 | −15 |

두 보정이 겹칠 수 있으므로 최종 점수는 기준 점수에 보정값의 합을 더한 뒤 0~100으로 클램핑합니다. 가장 잘 맞는 조합(상생 + 천간합 + 지지육합)은 85점, 최악의 조합(상극 + 지지충)은 10점이 됩니다.

### 탐색 피드 구성

사전 계산된 `compatibility_score`를 기반으로 탐색 피드를 내려줍니다. 기본적으로 비활성 유저, 이미 채팅 중인 상대, 신고한 유저는 항상 제외됩니다.

여기에 더해 유저가 `VERY_BAD`로 설정한 라이프스타일 항목은 피드에서 아예 걸러냅니다. `BAD`는 소프트 신호로 간주해 정렬 참고용으로만 활용하며 하드 필터에서는 제외합니다.

| 기피 항목 | 제외 조건 |
|-----------|----------|
| `SMOKING` | 흡연자(`SMOKER`) 제외 |
| `DRINKING` | 잦은 음주(`OFTEN`) 제외 |
| `EXERCISE` | 운동 안 함(`NEVER`) 제외 |
| `SAME_COLLEGE` | 동일 대학교 제외 |

`SAME_COLLEGE`는 파라미터 바인딩 방식으로 처리해 SQL 인젝션을 방지합니다. 최종 결과는 `compatibility_score DESC` 정렬 후 페이지네이션으로 반환됩니다.

ALB가 HTTP/WebSocket 트래픽을 로드밸런싱하고, ASG가 CloudWatch 지표에 따라 인스턴스를 자동 조정합니다. 각 인스턴스는 Docker로 Spring Boot + Nginx를 실행하며, RDS PostgreSQL과 ElastiCache Redis를 공유합니다. 프로필 사진은 CloudFront를 통해 S3에서 서빙됩니다.

## 💬 기술 — 실시간 채팅

### 멀티 서버 환경에서의 WebSocket

ALB가 트래픽을 여러 EC2 인스턴스로 분산하면 각 서버의 WebSocket 세션이 분리됩니다. 서버 A에 연결된 클라이언트가 서버 B의 세션으로 전달된 메시지를 수신하지 못하는 문제입니다. 이를 해결하기 위해 **Redis pub/sub**을 중간 브로커로 두었습니다. 메시지가 들어오면 해당 서버는 즉시 Redis 채널에 발행하고, 모든 서버가 채널을 구독하고 있다가 자신에게 연결된 클라이언트에게 STOMP로 전달합니다.

```
[서버 A에서 메시지 수신]
    → Redis에 publish("chat:room:{roomId}", message)
    → 모든 서버가 해당 채널을 subscribe하고 있음
    → 각 서버가 자신에 연결된 클라이언트에게 STOMP로 forward
```

### 24시간 미응답 자동 처리

150개 메시지에 도달하면 선택 대기 상태로 전환되고, Redis에 TTL 키를 설정합니다.

```
SET choice:expire:{roomId} 1 EX 86400
```

24시간 후 키가 만료되면 Redis keyspace notification이 발생하고, `ChoiceExpiryListener`가 이를 감지해 아직 `WAITING` 상태인 방을 자동으로 `REJECTED` 처리합니다. 별도 스케줄러 없이 Redis 만료 이벤트만으로 처리 시점을 보장합니다.

### 채팅방 상태 흐름

```
ACTIVE (0~149개 메시지)
    ↓ 150번째 메시지 전송
COMPLETED (선택 대기)
    ↓ 양쪽 선택 완료 또는 24h 만료
EXPIRED (종료)
```

| 결과 | 조건 |
|------|------|
| `REVEALED` | 둘 다 YES → 연락처 공개 |
| `REJECTED` | 하나라도 NO 또는 24h 미응답 |
